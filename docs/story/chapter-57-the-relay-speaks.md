# Chapter LVII — The Relay Speaks

*In which the fortress learns not only to broadcast intelligence outward, but to receive commands from beyond its walls — and the machines outside stop watching and start acting.*

> **Package:** This feature is in `titan_colossus` — add `import 'package:titan_colossus/titan_colossus.dart';` to use it.

---

The bridge had been a triumph.

The Terrain glowed on the AI scribes' desks now. The Stratagems appeared in their prompts. The dead ends were circled, the unreliable routes flagged, the Debrief summaries laid out in clean Markdown. Every machine that spoke the Model Context Protocol could see what the Scout had seen and what the Gauntlet had generated.

But they couldn't *do* anything about it.

"It's a one-way street," Fen said, pacing the workshop. On her desk, a stack of Campaign JSON sat in neat columns — tests generated by the Copilot overnight, brilliant edge cases targeting dead-end screens and failure recovery paths. "Copilot generated all of these. Nineteen Stratagems. They're perfect. And now..."

"And now someone has to paste them into the Lens UI," Rhea finished.

"Or load them from a file. Or write a script. Or—"

"Or we need a human," Kael said quietly.

The word hung in the air like a stone in a web. A human. The entire Blueprint pipeline had been built to remove the bottleneck of human intelligence. Scout analyzed passively. Gauntlet generated automatically. Campaign executed programmatically. But the final mile — getting the AI's Campaign from its mind into the running app — still required fingers on a keyboard.

"The bridge carries intelligence *from* us to them," Kael said. "We never built the road that carries orders *back*."

---

## The Relay

> *A fortress that can only shout outward is still half-deaf. The Relay teaches it to listen.*

Kael built it in a single night.

The concept was deceptively simple: embed a lightweight HTTP server *inside* the running Flutter app. Not a mock. Not a proxy. A real server, bound to a real port, accepting real requests from anything on the network that spoke HTTP.

He called it the **Relay**.

```dart
ColossusPlugin(
  enableRelay: true,
)
```

One parameter. One boolean. The fortress opened its ears.

Behind the scenes, the Relay bound a `dart:io` `HttpServer` to port 8642 on all network interfaces. It registered six endpoints — four for reading, two for writing. It wrapped every request in CORS headers so web-based clients could reach it. It logged every operation through Chronicle so the watchtower could track traffic.

And on the main isolate — the same event loop that drove the Flutter UI — it waited.

"Why the main isolate?" Fen asked, reading the architecture notes. "Isn't that risky? What if a campaign blocks the UI?"

"It *must* be the main isolate," Kael said. "The StratagemRunner synthesizes pointer events through `GestureBinding.handlePointerEvent`. That binding only exists on the main isolate. Campaign execution is async — it cooperates with the event loop, not fights it."

"And the HTTP server?"

"Also async. `dart:io`'s `HttpServer` runs on the same event loop. Incoming requests interleave naturally with frame callbacks and touch processing. The Relay doesn't steal time from the UI; it takes turns."

---

## The Endpoints

> *Six gates. Six purposes. Each one a conversation the fortress could never have before.*

```
GET  /health     — "Are you alive?"
GET  /status     — "What are your numbers?"
GET  /terrain    — "Show me the map."
GET  /blueprint  — "Tell me everything."
POST /campaign   — "Run these tests."
POST /debrief    — "Explain these results."
```

The `/health` endpoint required no authentication — it was the heartbeat, the first thing any client checked before committing to a longer conversation.

The other five required a bearer token:

```dart
ColossusPlugin(
  enableRelay: true,
  relayConfig: RelayConfig(
    port: 8642,
    authToken: 'my-secret-token',
  ),
)
```

"Without the token, any process on the network can execute campaigns in your app," Kael warned. "During development that's probably fine. In a CI pipeline, you want the lock."

When a request arrived without `Authorization: Bearer <token>`, the Relay returned 401 and logged the attempt through Chronicle. The fortress had ears, but it wasn't gullible.

## The Campaign Execution

> *The machines had always generated tests. Now they executed them too.*

The moment that changed everything was POST `/campaign`.

When the MCP server received an `execute_campaign` call — with the full Campaign JSON in the payload — it opened an HTTP connection to the Relay, posted the JSON, and waited.

Inside the app, the Relay parsed the request, extracted the Campaign body, and passed it to `Colossus.executeCampaignJson()`. The same method that ran campaigns from the Lens UI. The same StratagemRunner that synthesized taps and swipes and validated screen presence. The same Verdict pipeline that recorded every step.

```
AI → MCP Server (stdio) → HTTP POST → Relay (in-app) → Colossus
                                                ↓
                                        executeCampaignJson
                                                ↓
                                        CampaignResult → HTTP Response
```

The timeout was generous — ten minutes, by default — because campaigns involving twenty or thirty Stratagems needed time. Each Stratagem navigated between screens, waited for animations, verified element presence. Five hundred milliseconds per step, thirty steps per Stratagem, nineteen Stratagems. The math demanded patience.

When the campaign completed, the Relay serialized the result:

```json
{
  "campaign": "questboard-edge-cases",
  "passRate": 0.842,
  "totalExecuted": 19,
  "totalFailed": 3,
  "totalSkipped": 0,
  "report": "...",
  "aiDiagnostic": "...",
  "verdicts": [
    {
      "stratagem": "dead-end-escape-form-demo",
      "passed": false,
      "summary": {
        "totalSteps": 4,
        "passedSteps": 2,
        "failedSteps": 2,
        "failedRoutes": ["/form-demo"]
      }
    }
  ]
}
```

And sent it back upstream — HTTP response to MCP server to AI assistant.

The Copilot received the results. It saw the three failures. It read the `aiDiagnostic` section. It suggested code fixes. And then it called `execute_campaign` again to verify the fixes.

No human had touched a keyboard during any of it.

## The Cross-Platform Question

> *The fortress stood on many grounds. The Relay had to stand on all of them.*

"Does it work on iOS?"

Fen asked the question everyone was thinking. The Relay used `dart:io`'s `HttpServer`, which was native to Dart's runtime. But Dart's runtime differed by platform.

Kael pulled up the compatibility matrix:

| Platform | `dart:io` | HTTP Server | Relay Support |
|----------|-----------|-------------|---------------|
| Android  | ✅ | ✅ | Full |
| iOS      | ✅ | ✅ | Full |
| macOS    | ✅ | ✅ | Full |
| Windows  | ✅ | ✅ | Full |
| Linux    | ✅ | ✅ | Full |
| Web      | ❌ | ❌ | Stub |

"Five out of six," he said. "Web is the exception. Browser sandboxes can't host HTTP servers — it's a fundamental platform limitation, not a Dart limitation."

"What happens on web?"

"Nothing. Gracefully nothing." He showed the conditional import:

```dart
import 'relay_io.dart'
    if (dart.library.html) 'relay_stub.dart'
    as platform;
```

On web, the `RelayPlatform` stub provided the same API surface but every method was a no-op. `start()` completed immediately. `stop()` completed immediately. `status.isRunning` returned `false`. The app worked perfectly; it just couldn't receive HTTP commands.

"For web apps," Kael said, "you still paste Campaign JSON into the Lens Blueprint tab, or load it from a file. The Relay is for native platforms where a real server is possible — which is where most enterprise testing happens anyway."

Fen nodded. "Android emulators. CI runners on Linux. QA devices on the shelf."

"Exactly."

## The Distance Problem

> *The hardest part of a bridge isn't the span. It's knowing where the other end stands.*

When the MCP server ran on a developer's laptop and the app ran on the same laptop (desktop target or simulator), the Relay was trivially reachable at `localhost:8642`.

But when the app ran on a physical Android device connected via USB, `localhost` pointed to the wrong machine.

Kael documented the solution for each scenario:

**Android emulator or device via USB:**
```bash
adb forward tcp:8642 tcp:8642
```

One command. The Android debug bridge forwarded the laptop's port 8642 to the device's port 8642. Now `localhost:8642` on the laptop reached the Relay inside the app on the phone.

**iOS simulator:** `localhost` worked directly — the simulator shared the host's network stack.

**iOS device:** Use the device's WiFi IP address. The Relay bound to `0.0.0.0` (all interfaces) by default, making it reachable from any device on the same network.

The MCP server accepted connection parameters via CLI arguments:

```json
{
  "github.copilot.chat.mcpServers": {
    "titan-blueprint": {
      "command": "dart",
      "args": [
        "run", "titan_colossus:blueprint_mcp_server",
        "--relay-host", "127.0.0.1",
        "--relay-port", "8642",
        "--relay-token", "my-secret"
      ],
      "cwd": "${workspaceFolder}/packages/titan_colossus"
    }
  }
}
```

## The Silence of Full Automation

That evening, Rhea ran the full cycle for the first time.

She launched the Questboard on her Android test device. The ColossusPlugin initialized Colossus, started the Shade listener, registered the Lens tabs, and — now — started the Relay.

```
ℹ [Relay] Relay started on 0.0.0.0:8642
ℹ [Relay] Auth token: debug-test-token
```

She opened her IDE. Typed a message to Copilot:

> "Test the entire Questboard navigation flow. Run the tests on the live app and tell me what fails."

Copilot called `relay_status`. The Relay was running.

Copilot called `get_full_context`. Seven screens, eleven transitions, one dead end at `/form-demo`, nineteen Stratagems already generated by the Gauntlet.

Copilot generated a Campaign. Twenty-three entries — four critical-path traversals and nineteen edge-case probes. It called `execute_campaign` with the full Campaign JSON.

Inside the app, the StratagemRunner took over. Synthetic taps navigated from screen to screen. The UI animated. Scroll positions changed. Text fields received input. Back buttons were pressed. Error screens were verified. For two minutes and forty seconds, the app tested itself while the phone sat untouched on Rhea's desk.

The results came back:

```
# Campaign Execution Results

**Pass Rate**: 82.6%

## Verdicts

✅ **critical-path-quest-lifecycle** — PASSED
✅ **critical-path-settings-roundtrip** — PASSED
✅ **critical-path-all-screens-reachable** — PASSED
❌ **critical-path-dead-end-form-demo** — FAILED
  - Navigate back from /form-demo: No back navigation found
✅ **bedrock-back-home** — PASSED
✅ **bedrock-back-quest-list** — PASSED
...
❌ **forgotten-outpost-form-demo** — FAILED
  - Verify /form-demo has outgoing navigation: Dead end confirmed
```

Copilot analyzed the failures. The `/form-demo` screen had no back button and no outgoing navigation. It suggested adding a `BackButton` widget and a route from `/form-demo` back to `/home`.

Rhea looked at her phone, still sitting on the desk. She hadn't touched it once.

"It works," she said.

Fen looked over. "What works?"

"All of it. The AI asked the app to test itself. The app tested itself. The AI read the results and told me what to fix."

The workshop was quiet for a long moment.

"That's the goal, isn't it?" Kael said from behind his screen. "The machines test. The machines diagnose. We fix. The machines verify the fix. The humans write features. The machines verify the features."

He paused.

"The Relay isn't a server. It's the last wall coming down."

## The Five Bridges

Kael updated the architecture diagram one final time. The four bridges from the previous chapter had become five:

```
              ┌──────────────────────────────────┐
              │           War Room               │
              │  Scout · Terrain · Gauntlet      │
              │  Campaign · Verdict · Debrief    │
              │           Relay (HTTP)           │
              └──┬──────┬──────┬──────┬──────┬───┘
                 │      │      │      │      │
      ┌──────────┤      │      │      │      ├────────┐
      ▼          ▼      ▼      ▼      ▼      ▼        ▼
 Auto-Export   CLI   MCP Read  Lens  MCP Execute  Relay HTTP
 (onDetach)  (CI)   (stdio→)  (UI)  (stdio←→)   (in-app)
      │          │      │      │      │            │
      └──────────┴──────┴──────┴──────┴────────────┘
                        │
                   AI Assistants
              (Copilot · Claude · etc.)
```

The first four bridges carried intelligence *outward* — from the app to the AI. The fifth bridge, the Relay, carried commands *inward* — from the AI to the app. Together, they formed a complete circuit. A closed loop. A system where the only human involvement was deciding what to build next.

**New in this chapter:**
- `Relay` — Embedded HTTP server in the running Flutter app for AI-driven campaign execution
- `RelayConfig` — Configuration: port (default 8642), host, authToken, requestTimeout, enableLogging
- `RelayStatus` — Runtime status: isRunning, port, host, requestsHandled, campaignsExecuted, startedAt
- `RelayHandler` — Interface for delegating requests to Colossus (executeCampaign, getTerrain, getBlueprint, debriefVerdicts)
- `ColossusPlugin(enableRelay: true)` — One-parameter Relay activation
- `ColossusPlugin(relayConfig: RelayConfig(...))` — Custom Relay configuration
- `Colossus.instance.startRelay()` — Manual Relay startup
- `Colossus.shutdownAsync()` — Async shutdown that awaits Relay cleanup
- Conditional import: `relay_io.dart` (5 platforms) / `relay_stub.dart` (web no-op)
- Endpoints: `/health`, `/status`, `/terrain`, `/blueprint`, `/campaign`, `/debrief`
- MCP tools: `execute_campaign`, `relay_status`, `relay_terrain`
- CLI args: `--relay-host`, `--relay-port`, `--relay-token`
- Bearer token authentication with CORS support
